# AI Solver Spec

AI Solver 是核心产品能力，不是聊天页。它必须围绕一道物理竞赛题、标准答案和结构化学习解析工作。

## Validation Flow

```text
create session
-> upload materials
-> server validates problem material and standard-answer material both exist
-> Gemini/OpenRouter reads supported multimodal materials directly
-> run structured AI analysis
-> show fixed collapsible sections
```

## Upload Boundary

Supported：

- images：1 到 10 张。
- PDF：1 个。
- DOCX：1 个。

Not supported in v1 validation：

- old `.doc`
- pure text-first input
- automatic multi-problem splitting
- running final analysis when problem or standard-answer material is missing

Material roles：

- `problem`
- `answer`
- `combined`

## Server-side Gate

```text
No standard answer, no AI solution.
```

UI disabled state is not enough. The analyze route must reject requests unless the session owns both problem material and standard-answer material. Combined legacy materials may still satisfy both sides when explicitly used by server-side workflows.

## AI Provider Architecture

Current provider：Gemini 3 through OpenRouter。

OpenRouter is OpenAI-compatible, so the server provider layer should use a chat completion abstraction:

```text
UI
-> server route / server action
-> AiSolverService
-> RetrievalService
-> AiAnalysisProvider
-> OpenAI-compatible ChatCompletionClient
-> OpenRouter adapter
```

Environment variables：

- `OPENROUTER_API_KEY`：server-side only。
- `OPENROUTER_MODEL`：model id config，do not hardcode in UI。
- optional `OPENROUTER_BASE_URL`。

Provider rules：

- UI must not import provider code。
- No `NEXT_PUBLIC_OPENROUTER_API_KEY`。
- Provider raw errors must be normalized before reaching client。
- Logs must not include secrets。
- Future providers can include DeepSeek、Qwen、Doubao、Kimi、GLM without rewriting page flow。

### Multi-stage generation contract

Structured analysis now follows a three-stage orchestration contract：

```text
1. server planning
   -> confirm the materials gate
   -> instantiate the fixed seven-section task template with <= 5000 Chinese character budgets

2. per-task generation calls
   -> generate one section-oriented task at a time
   -> every task declares target_char_limit <= 5000 Chinese characters

3. server aggregation
   -> deterministically merge validated task outputs back into the fixed final JSON schema
```

Detailed prompt language, section mappings, and schema expectations live in
`docs/ai-solver-prompt-design.md`. That document also records richer provider-side
planning and aggregation prompt designs for future experimentation, while the current runtime
uses deterministic server planning + deterministic server assembly to avoid extra structured-output failure points.

## Structured Output Sections

AI output must map into these fixed fields：

1. `step_by_step_derivation`
2. `physical_reasoning_reconstruction`
3. `related_models_similar_problems`
4. `related_articles`
5. `key_handling`
6. `write_article`
7. `add_to_personal_library`

UI must render sections as collapsible panels.

## Retrieval Rules

Similar problems：

- model classification explanation may be generated by AI。
- actual similar problem records must come from Problem Bank retrieval。
- if retrieval is not connected, return empty list and `not_connected` status。

Related articles：

- actual article records must come from Article Plaza retrieval。
- if retrieval is not connected, return empty list and `not_connected` status。

Forbidden：

- fabricated problem ids
- fabricated article titles
- fabricated authors
- fabricated source papers

## Server Data Requirements

Current tables：

- `ai_solver_sessions`
- `uploaded_materials`
- `ai_solution_outputs`
- `ai_messages` for follow-up Q&A

All private records must include `user_id` and use owner-only RLS.

注：`extracted_materials` 已不再需要，当前 Gemini 直接读取 multimodal 材料。

## v1 已实现与待实现

已实现：
- ✅ follow-up Q&A（threaded + selected-text）
- ✅ multimodal image/PDF input

待实现：
