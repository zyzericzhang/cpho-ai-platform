# AI Solver Spec

AI Solver 是核心产品能力，不是聊天页。它必须围绕一道物理竞赛题、标准答案和结构化学习解析工作。

## Validation Flow

```text
create session
-> upload materials
-> extract text
-> user confirms/edits problem text, diagram notes, standard answer
-> server validates confirmed standard answer exists
-> run structured AI analysis
-> show fixed collapsible sections
```

## Upload Boundary

Supported：

- images：1 到 10 张。
- PDF：1 个。
- DOCX：1 个。

Not supported in v1 validation：

- old `.doc`
- pure text-first input
- automatic multi-problem splitting
- sending original images/PDF/DOCX directly to final analysis provider

Material roles：

- `problem`
- `answer`
- `combined`

## Confirmation Fields

User must confirm or edit：

- `problem_text`
- `diagram_notes`
- `standard_answer`

Server-side gate：

```text
No standard answer, no AI solution.
```

UI disabled state is not enough. The analyze route must reject empty or unconfirmed standard answer.

## AI Provider Architecture

Current provider：Gemini 3 through OpenRouter。

OpenRouter is OpenAI-compatible, so the server provider layer should use a chat completion abstraction:

```text
UI
-> server route / server action
-> AiSolverService
-> RetrievalService
-> AiAnalysisProvider
-> OpenAI-compatible ChatCompletionClient
-> OpenRouter adapter
```

Environment variables：

- `OPENROUTER_API_KEY`：server-side only。
- `OPENROUTER_MODEL`：model id config，do not hardcode in UI。
- optional `OPENROUTER_BASE_URL`。

Provider rules：

- UI must not import provider code。
- No `NEXT_PUBLIC_OPENROUTER_API_KEY`。
- Provider raw errors must be normalized before reaching client。
- Logs must not include secrets。
- Future providers can include DeepSeek、Qwen、Doubao、Kimi、GLM without rewriting page flow。

## Structured Output Sections

AI output must map into these fixed fields：

1. `step_by_step_derivation`
2. `physical_reasoning_reconstruction`
3. `related_models_similar_problems`
4. `related_articles`
5. `key_handling`
6. `write_article`
7. `add_to_personal_library`

UI must render sections as collapsible panels.

## Retrieval Rules

Similar problems：

- model classification explanation may be generated by AI。
- actual similar problem records must come from Problem Bank retrieval。
- if retrieval is not connected, return empty list and `not_connected` status。

Related articles：

- actual article records must come from Article Plaza retrieval。
- if retrieval is not connected, return empty list and `not_connected` status。

Forbidden：

- fabricated problem ids
- fabricated article titles
- fabricated authors
- fabricated source papers

## Server Data Requirements

Initial tables planned：

- `ai_solver_sessions`
- `uploaded_materials`
- `extracted_materials`
- `ai_solution_outputs`
- `ai_messages` for future follow-up Q&A

All private records must include `user_id` and use owner-only RLS.

## v1 Validation Does Not Require

- full history persistence as launch blocker
- follow-up Q&A
- article creation from AI output
- real retrieval implementation
- multimodal final analysis

But the flow must leave room for these later.
